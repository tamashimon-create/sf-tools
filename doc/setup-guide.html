force-* プロジェクト 新規セットアップガイド

新しい Salesforce プロジェクト（force-xxx）を GitHub に作成し、sf-tools と連携させるまでの手順

!
前提条件
⚠️
Windows ユーザーへ: 本ガイドのすべての操作は Git Bash で行ってください。コマンドプロンプトや PowerShell は非対応です。

ローカル環境

Git（Git Bash 含む） ソースコード管理
GitHub CLI（gh） GitHub 操作の自動化
Node.js / npm ビルドツール・依存管理
Salesforce CLI（sf） Salesforce 組織との接続・デプロイ
Visual Studio Code Salesforce Extension Pack (Expanded) を事前にインストール
Slack ワークフローの通知先

Salesforce 環境

  • 開発用の Sandbox（Developer / Developer Pro）が作成済みであること
  • 接続先の Sandbox 名を把握していること

GitHub アカウント

  • リポジトリ作成権限があること
0
環境の動作確認

セットアップを始める前に、必要なツールが正しくインストールされているか確認します。

# 各ツールのバージョンを確認
git --version        # Git
gh --version         # GitHub CLI
node --version       # Node.js
npm --version        # npm
sf --version         # Salesforce CLI
code --version       # Visual Studio Code
✔️
すべてバージョンが表示されれば OK。エラーが出る場合は、該当ツールのインストールをやり直してください。

GitHub CLI の認証確認:

gh auth status
✔️
「Logged in to github.com」と表示されれば OK。未認証の場合は gh auth login を実行してください。
1
sf-tools をクローン
git clone https://github.com/tama-create/sf-tools.git ~/sf-tools

PATH に追加して、どこからでもスクリプトを実行できるようにします:

echo 'export PATH="$HOME/sf-tools:$PATH"' >> ~/.bashrc
source ~/.bashrc
2
プロジェクトを作成

テンプレートから GitHub リポジトリを作成し、ローカルにクローンします。

# <ユーザー名> は自分の GitHub ユーザー名または組織名に置き換える
gh repo create <ユーザー名>/force-xxx --template tama-create/force-template --private --clone
cd force-xxx
⚠️
sf project generate は使わないでください。プロジェクト構成・ドットファイルはすべてテンプレートに含まれています。
3
sf-start.sh で初期セットアップ
sf-start.sh

初回実行時は接続先を聞かれるので、開発用 Sandbox の名称を入力します。ブラウザが開くのでログインしてください。以降は自動接続されます。

初回実行で自動的に行われること

  • Salesforce 組織への接続 & VS Code 起動
  • GitHub Actions ワークフロー（wf-*.yml）の生成
  • 設定ファイル（branches.txt / metadata.txt）の生成
  • Git フック・マージドライバーの設定
4
ブランチ構成を選択
sf-branch.sh

実行すると、以下の構成から選択できます:

1
main / staging / develop 開発 → 検証 → 本番の3段階リリース
2
main / staging 検証 → 本番の2段階
3
main 小規模プロジェクト・単独開発向け

選択すると、必要なブランチが GitHub に自動作成されます。

5
GitHub Secrets を設定

以下の Secret を gh secret set コマンドで登録します。プロンプトが表示されるので、値を貼り付けて Enter で確定してください。

認証 URL の取得と登録

各 Salesforce 組織にログインして認証 URL を取得し、Secret に登録します。
組織ごとに「ログイン → URL 取得 → Secret 登録」を繰り返します。

本番組織（必須）

# 本番は login.salesforce.com（デフォルト）なので --instance-url 不要
sf org login web --alias prod
sf org display --verbose --json --target-org prod | grep sfdxAuthUrl
gh secret set SFDX_AUTH_URL_PROD -R <ユーザー名>/force-xxx

ステージング組織（2階層以上の場合）

sf org login web --alias staging --instance-url https://test.salesforce.com
sf org display --verbose --json --target-org staging | grep sfdxAuthUrl
gh secret set SFDX_AUTH_URL_STG -R <ユーザー名>/force-xxx

開発組織（3階層の場合）

sf org login web --alias develop --instance-url https://test.salesforce.com
sf org display --verbose --json --target-org develop | grep sfdxAuthUrl
gh secret set SFDX_AUTH_URL_DEV -R <ユーザー名>/force-xxx

PAT_TOKEN の作成と登録

ワークフローがブランチ保護をバイパスして push するために必要です。

  1. GitHub にログインし、右上のアイコン → Settings を開く
  2. 左メニューの一番下 Developer settings を開く
  3. Personal access tokens → Fine-grained tokens を開く
  4. Generate new token をクリック
  5. 以下を設定する:
    項目設定値
    Token namesf-metasync
    ExpirationNo expiration
    Repository access「Only select repositories」→ 対象の force-xxx を選択
    Permissions → Repository permissions → ContentsRead and write
  6. Generate token をクリックし、表示されたトークンをコピー
⚠️
トークンはこの画面でしか表示されません。必ずコピーしてから画面を閉じてください。
gh secret set PAT_TOKEN -R <ユーザー名>/force-xxx
# プロンプトが表示されたら、コピーしたトークンを貼り付けて Enter

SLACK_BOT_TOKEN の取得と登録

  1. Slack API にアクセス
  2. Create New App → From scratch を選択
  3. App 名（例: SF Deploy Bot）とワークスペースを選択して作成
  4. 左メニューの OAuth & Permissions を開く
  5. Scopes セクションまでスクロールし、Bot Token Scopes に chat:write を追加
  6. ページ上部の Install to Workspace をクリック → 「許可する」
  7. 表示される Bot User OAuth Token（xoxb- で始まる）をコピー
gh secret set SLACK_BOT_TOKEN -R <ユーザー名>/force-xxx
# プロンプトが表示されたら、コピーしたトークンを貼り付けて Enter

SLACK_CHANNEL_ID の取得と登録

  1. Slack で通知先のチャンネルを開く
  2. チャンネル名をクリック → 詳細パネルの最下部に「チャンネル ID」が表示される（C で始まる文字列）
  3. 作成した Bot をこのチャンネルに招待する（チャンネルで /invite @SF Deploy Bot）
gh secret set SLACK_CHANNEL_ID -R <ユーザー名>/force-xxx
# プロンプトが表示されたら、コピーした ID を貼り付けて Enter
📋
Secret 一覧チェックリスト:
  • SFDX_AUTH_URL_PROD — 本番組織の認証URL（必須）
  • SFDX_AUTH_URL_STG — ステージング組織の認証URL（2階層以上）
  • SFDX_AUTH_URL_DEV — 開発組織の認証URL（3階層）
  • PAT_TOKEN — Fine-grained トークン（必須）
  • SLACK_BOT_TOKEN — Slack Bot Token（必須）
  • SLACK_CHANNEL_ID — Slack チャンネル ID（必須）
6
初回コミット & プッシュ

Step 3〜4 で生成されたファイルをまとめてコミットします。

# .env や認証情報が含まれていないことを確認
git status

git add -A
git commit -m "chore: sf-tools 初期セットアップ"
git push origin main
⚠️
git status で .env やパスワードファイルが表示されていないことを必ず確認してからコミットしてください。
7
リポジトリの Ruleset を設定
⚠️
Ruleset は status checks を要求するため、必ず Step 6 のプッシュ後に設定してください。

スクリプトで自動設定（推奨）:

# <ユーザー名>/force-xxx は自分のリポジトリに置き換える
repo-settings.sh <ユーザー名>/force-xxx
手動で設定する場合（Settings → Rules → Rulesets）

protect-main（main ブランチ）

  • 対象: main
  • ルール:
    • 削除禁止
    • 強制プッシュ禁止
    • Required status checks:
      • call / マージ順序を検証（プロモーション順序チェック）
      • call / Salesforce 検証（本番反映なし・確認のみ）
  • Bypass actors: Repository admin

protect-staging（staging ブランチ）※2階層以上の場合

  • 対象: staging
  • ルール: protect-main と同様
8
動作確認

sf-metasync の確認

GitHub → Actions → 「Salesforce メタデータ自動同期」→ 「Run workflow」で手動実行します。

✔️
正常に完了すれば本番組織のメタデータが main に同期されます。

PR ワークフローの確認

テスト用のブランチを作成して PR を出します:

git checkout -b feature/test-setup
git commit --allow-empty -m "test: ワークフロー動作確認"
git push -u origin feature/test-setup
sf-next.sh

sf-next.sh がブランチ構成に応じた PR 先を自動判定し、ブラウザで PR 作成画面を開きます。

PR が作成されると以下が自動実行されます:

  • wf-validate（デプロイ検証）— すべての PR で実行
  • wf-sequence（マージ順序チェック）— main / staging への PR で実行
⚙
ワークフロー一覧（自動生成済み）
ファイルトリガー目的
wf-validate.ymlPR 作成・更新時dry-run 検証 → PR にコメント
wf-sequence.ymlmain/staging への PR 時マージ順序チェック
wf-release.ymlPR マージ後対応組織へリリース → Slack 通知
wf-propagate.ymlmain への PR マージ後staging / develop へ伝播
wf-metasync.yml平日毎時（JST 9:00〜19:00）本番メタデータを main へ同期
📁

トラブルシューティング
sf-metasync が「push declined due to repository rule violations」で失敗
PAT_TOKEN が未設定、または権限不足です。Step 5 の PAT_TOKEN 設定を確認してください。
wf-validate / wf-sequence が動かない
ワークフローファイルが .github/workflows/ にあるか確認してください。sf-start.sh を再実行すれば未存在のワークフローは自動コピーされます。
CRLF の警告が出る
Windows 環境での改行コードの違いです。動作に影響はありません。git checkout -- <file> でリセット可能です。
pre-push フックがプッシュをブロックする
main ブランチへの直接プッシュは禁止されています。PR 経由でマージしてください。テスト等でバイパスが必要な場合は git push --no-verify を使用してください。
