# sf-tools 設計思想とビジネス差別化

> 本書は sf-tools の設計哲学・競合との差別化・AI 時代における位置づけをまとめた内部ドキュメントです。
> 実装詳細（ファイルパス・コマンド等）は各スクリプトおよび `sf-cicd-strategy.md` を参照してください。

---

## 1. なぜ DevOps Center ではないのか

### 1.1 DevOps Center の本質的な限界

Salesforce DevOps Center は「Git を知らない Admin でも変更管理できるように GUI 化した」ツールである。
この設計思想が、そのまま弱点になっている。

| 場面 | DevOps Center | sf-tools |
|---|---|---|
| 通常のリリース | GUI でポチポチ操作 | `sf-deploy` 一発 |
| コンフリクト発生 | GUI では解決不能 → Git 職人が登場 | 最初から Git で管理しているため普通に解決できる |
| 複数人の並行開発 | ブランチ管理が GUI 前提で破綻しやすい | PR ベースのワークフローがそのまま使える |
| CI（自動テスト） | DevOps Center 単体では完結しない | GitHub Actions と直結している |
| 新メンバーのオンボード | GUI の操作を覚えさせる必要がある | `sf-init` 一発でセットアップ完了 |

DevOps Center はコンフリクトが**起きてから**どう解決するかを前提とした設計になっている。
sf-tools は**コンフリクトが起きにくい構造を最初から作る**設計になっている。

### 1.2 「隠す」vs「融合する」

**DevOps Center の発想:**
> Git の複雑さを GUI で隠す → 使いやすくなる（はず）

**sf-tools の発想:**
> Git・GitHub・VSCode・Slack をそれぞれ得意なことで使い、**継ぎ目をなくす**

後者の方がエンジニアとして正しい。ツール本来の力を殺さずに使い倒せる。
見た目の派手さより、**案件起案（ブランチ作成）から開発・テスト・リリースまでの一気通貫**を重視する。

### 1.3 DevOps Center の需要の実態

DevOps Center のようなツールは **Salesforce 特有の需要** である。
一般的なスクラッチ開発・モダンな開発現場では、GUI でブランチ管理するという発想自体が生まれない。

| 対象 | 需要 |
|---|---|
| Git を知らない Salesforce Admin | ある（DevOps Center の本来のターゲット） |
| Salesforce 専業の小規模チーム | 微妙（習得コストに見合うか疑問） |
| エンジニアが主体の開発チーム | ほぼない |
| 一般的な Web・スクラッチ開発 | 完全にない |

sf-tools と DevOps Center は**土俵が根本的に異なる**。対象層が違うため競合ではない。
むしろ「DevOps Center では物足りなくなったチーム」が sf-tools に移行する流れもあり得る。

## 2. sf-tools の設計哲学

### 2.1 Git ネイティブ × ツール透明化

sf-tools の中心的な設計思想：

> **「Git の概念は理解する。Git の操作はツールに任せる。」**

これは矛盾ではない。車の運転に例えると、エンジンの仕組みをすべて知らなくても運転はできる。
しかし「アクセルとブレーキの意味」を知らないと話にならない。

Git コマンドを直接打たない設計になっているが、Git の概念（ブランチ・コミット・PR・マージ）は
利用者が理解している前提で設計されている。これが DevOps Center の「概念ごと隠す」設計との違いである。

```
sf-push    → git add / commit / push を隠蔽
sf-next    → どのブランチに PR を出すべきか案内
sf-start   → org 接続 + ブランチ確認 + VSCode 起動
pre-push   → 自動 dry-run（知らないうちに守られている）
```

Salesforce 開発者が意識するのは「今日は○○機能を作った → sf-push → PR を出す」だけでよい。
**Git が透明になっている**。

### 2.2 コンフリクト予防の仕組み

これは提案時の重要な訴求ポイントである。

- **pre-push フックで master からマージ** → 「プッシュしたら壊れた」を事前排除
- **sf-metasync で本番の最新状態を定期取得** → 本番と開発の乖離を最小化
- **ブランチ構成を sf-init で統一** → 全員が同じルールで動く

「コンフリクトが起きたらどう解決するか」ではなく、「そもそも起きにくい構造を作る」という思想を
プロジェクト開始時点から組み込んでいる。

### 2.3 案件起案からリリースまで一気通貫

sf-tools は Salesforce 開発の全フローをカバーしている。

```
ブランチ作成      sf-branch（ブランチ構成の整備）
        ↓
開発開始          sf-start（org 接続 + VSCode 起動）
        ↓
コミット前確認    pre-push フック（dry-run 自動実行）
        ↓
プッシュ          sf-push（master マージ込み）
        ↓
PR 管理           sf-next（次の PR 先を案内）
        ↓
リリース          sf-deploy / sf-release（GitHub Actions 連動）
        ↓
メタ同期          sf-metasync（本番差分を自動取得・commit・PR）
```

Git・GitHub・VSCode・Slack をそれぞれ得意な役割で使い、
開発者は**ツールの継ぎ目を意識しない**で作業できる。

## 3. AI 時代との親和性

### 3.1 バイブコーディングと Git の必要性

AI とのペアプログラミング（バイブコーディング）を経験すると、Git の重要性が逆説的に上がる。

```
従来：エンジニアが一人でコードを書く → Git は個人の作業ログ

AI 時代：Claude と一緒に作る → Git がないと
            ・どこまで AI に任せたか分からない
            ・壊れたときに戻れない
            ・何をレビューすればいいか分からない
```

「AI が書いてくれるから Git はいらない」ではなく、**「AI が書くからこそ Git が必須」** になる。
Git は AI 時代においてむしろ重要度が増すインフラである。

### 3.2 「Git コマンドを打たない」設計の意味

Salesforce の開発者には、もともとエンジニアでない人も多い。
そうした開発者に必要なのは「Git の完全な習熟」ではなく「Git の概念理解と安全な操作環境」である。

sf-tools はその環境を提供する：

- **Git の概念**（ブランチ・PR・マージ）は理解している前提
- **Git のコマンド操作**はツールが代行する
- **深い Git 知識**（コンフリクト解消・rebase 等）はプロジェクト管理者だけが持てばよい

この設計は、AI が Salesforce 開発の現場に入ってくる未来においても有効である。
Claude と一緒に開発する開発者が増えるほど、「Git を透明化した安全な作業環境」の価値は上がる。

### 3.3 AI 時代における sf-tools の位置づけ

| | DevOps Center | sf-tools |
|---|---|---|
| Git の扱い | GUI で隠す（概念も隠す） | 操作を隠す（概念は理解させる） |
| コンフリクト発生時 | お手上げ | Git が分かる人が普通に解決できる |
| AI との親和性 | 低い（GUI 前提） | 高い（Git ネイティブ） |
| 未来への対応力 | 厳しい | 伸びしろがある |

## 4. 日本市場での差別化

### 4.1 CI/CD 導入状況の実態

日本の Salesforce 開発市場における CI/CD 導入状況（2025年時点の推定）：

| 層 | CI/CD 提供状況 |
|---|---|
| 外資コンサル（Accenture・Deloitte 等） | 標準提供（グローバル CoE から展開。顧客は大企業限定） |
| テラスカイ等の技術系 SIer | 技術はある・全案件での標準提供ではない |
| 中堅 Salesforce 専業 SIer | 多くは未整備。属人的な手動デプロイが主流 |
| 中小 SIer・フリーランス | ほぼ未対応 |

**「Salesforce 開発で CI/CD まで整備してくれる会社は少ない」は、現時点では概ね事実である。**
特に中小〜中堅 SIer においては、CI/CD 提案は加点程度の扱いになっている。

### 4.2 訴求できる層・できない層

**効きやすい相手：**
- 内製化を進めようとしている事業会社（Salesforce 担当チームを抱え始めた企業）
- リリースサイクルが速い・Sandbox 数が多い案件（金融・流通・SaaS 系）
- 手動デプロイのヒヤリハット経験がある担当者

**効きにくい相手：**
- 大手外資コンサルと競合する案件（同等以上の体制がすでにある）
- 年 1〜2 回しかリリースしない案件（ROI が見えにくい）

### 4.3 差別化の賞味期限と展望

Salesforce 純正の DevOps Center が進化中のため、2〜4 年以内に「CI/CD は当たり前」になる可能性がある。
ただし以下の問題は残り続ける：

- DevOps Center は GitHub 連携があるが、ワークフロー設計・テスト自動化・JWT 管理は自前
- 「ツールが整っても使いこなせる人がいない」という問題は DevOps Center では解決しない
- **「新規プロジェクトの初期構築（sf-init）から日常運用（sfl ランチャー）まで一気通貫で面倒を見る」** という価値は DevOps Center では提供できない

競合が「ツール導入」どまりのところ、**「使えるようになるまで面倒を見る」** のが差別化のポイントになる。

### 4.4 提案時のポジショニング

> **「エンジニアが使い慣れたツール（Git / GitHub / VSCode / Slack）の上に、
> Salesforce 開発のベストプラクティスを乗せた基盤」**

- DevOps Center の代替ではなく、**Git ネイティブで Salesforce 開発を回せるチーム向けの基盤**
- Salesforce 開発案件に sf-tools をセットで提供することで、**CI/CD 環境込みの提案**が可能になる
- 訴求先は中堅 SIer・内製化を進める事業会社が最も有効

---

## 5. まとめ

sf-tools が目指すのは、**見た目の派手さではなく、現場で確実に動く一気通貫の開発基盤**である。

1. Git ネイティブで設計されているため、AI 時代の開発スタイルと親和性が高い
2. コンフリクト予防の仕組みを最初から組み込んでいるため、属人化しにくい
3. Git コマンドを打たない設計により、Salesforce 開発者のスキル差を吸収できる
4. DevOps Center とは対象層が異なるため、直接競合せず、むしろ補完関係にある
5. 日本市場では「CI/CD まで整備して提案できる会社」はまだ少なく、差別化の機会がある
