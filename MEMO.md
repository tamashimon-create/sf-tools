# Salesforce Git 運用・CI/CD 仕様書（確定版 Ver.3）

## 1. 目的
Salesforce メタデータの「パッチ形式」の特性を最大限に活かし、Git と各組織（Production/Staging/Dev）の整合性を保つ。特に、本番組織の変更を **SGD (Salesforce Git Delta)** を用いて高速に検知し、人間を介さず全環境へ即座に波及させる「自己修復型同期」を実現する。

## 2. 環境構成とブランチ構造

| 層 | 環境（Org） | ブランチ | 役割 |
| :--- | :--- | :--- | :--- |
| **本番層** | Production | `main` | **正（Source of Truth）**。本番の最新状態。 |
| **検証層** | Staging / UAT | `staging` | 結合テスト・顧客確認用。 |
| **開発層** | Development / QA | `develop` | 初期検証用。開発のベースライン。 |
| **作業層** | 各 Sandbox | `feature/*` | 作業項目ごとのソースの源泉。 |

---

## 3. リリースおよびマージ制限

### 3.1 スター型プロモート（配布ルール）
すべてのリリースは `feature/*` ブランチを起点とし、各ターゲットブランチへ個別に PR（プルリクエスト）を作成してデプロイする。
* **許可されるパス:**
    * `feature/*` → `develop`
    * `feature/*` → `staging`
    * `feature/*` → `main`
* **デプロイ方法:** 開発者が `feature/*` ブランチ内に作成した `package.xml` に基づき、修正分のみをリリースする。

### 3.2 マージ禁止制限（システムによる強制）
環境の独立性を保つため、以下のブランチ間の直接マージは GitHub Actions で検知し、エラーとしてブロックする。
* ❌ `develop` → `staging`
* ❌ `staging` → `main`
* ❌ `develop` → `main`

---

## 4. 本番組織の変更検知と完全自動同期（SGD活用）

### 4.1 本番チェックの自動化
本番組織（Production）での変更（リリースおよび手動修正）を Git に反映するプロセスから人間の介在を排除する。

* **実行スケジュール:** 平日 9:00〜19:00 (JST) の間、1時間おきに実行。
* **高速化技術:** **Salesforce Git Delta (SGD)** を使用。本番から取得したメタデータと `main` ブランチを比較し、差分のみを抽出。
* **完全自動マージ:**
    1. GitHub Actions が本番から最新メタデータを `retrieve`。
    2. 差分が検知された場合、システムが `main` ブランチへ **Git コマンドにより直接プッシュ** を行う（PRは不要）。

### 4.2 下流ブランチへの自動連鎖マージ（バックプロモート）
`main` ブランチが更新（自動同期または PR マージ）された直後、以下の連鎖マージを自動実行する。

1. **`main` → `staging`**: `main` の最新変更を `staging` へマージ。
2. **`main` → `develop`**: `main` の最新変更を `develop` へマージ。

**※注意:** このマージは Git 上の同期のみであり、Sandbox 組織への自動デプロイは行わない。開発者は必要に応じて自身の Sandbox へ最新状態を反映する。

## 5. 品質ガードレール（CI/CD）

### 5.1 ローカル：Git Hook (pre-push)
* `git push` 前に、リモートの最新 `main`（自動更新された本番の状態）を取り込んでいるかチェック。未取り込みの場合はプッシュを阻止し、コンフリクトを未然に防ぐ。

### 5.2 リモート：GitHub Actions
* **検証 (Validation):** PR 作成時に `sf project deploy start --dry-run` を実行。
* **通知機能:** 本番同期ジョブや自動マージが失敗（コンフリクト等）した際、即座に Slack/Teams 等のチャットツールへ通知を飛ばし、人間が介入できるようにする。

---

## 6. 運用上のメリット
1. **管理コストの削減:** 本番の同期に PR や承認を必要としないため、管理者の負担が最小限になる。
2. **デグレの防止:** 本番の修正が 1時間以内に開発環境（develop）まで届くため、古いコードをベースにした開発を防げる。
3. **リリースの柔軟性:** 変更セットのように「必要なものだけを、必要な環境に」直接送ることができ、かつ Git で履歴を完璧に管理できる。
